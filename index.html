Maphub Open Annotation API Documentation

20 July 2012

Abstract

This document describes how Maphub provides a Linked Data API by implementing the W3C Open Annotation specification. The provided examples depict how to use the Open Annotation model in the context of digitized historic maps or high-resolution image material and also show how to express Semantic Tags. The Maphub API currently provides access to two types of annotations:

  • Georeference Annotation: users create an annotation with the intention to express a correspondence between a point/region on the map and a point/region in a defined geographic coordinate system or an authoritative Gazetteer. The goal of this type of annotation is to provide a georeference for the map.
  • Commentarial Annotation: users associate textual information, tags, links to thesaurus terms or Semantic Web resources, etc. with a point or region on the map. The goal of this type of annotation is to provide additional information about the map or a map segment.

The definitions and examples in this document refer to Maphub version 0.3.

1. Introduction

Historic maps record historical geographical information and give insight into socio-economic and environmental phenomena such as land use, river channel changes or flood. They allow the reconstruction of past urban environments and draw a picture of the cultural, political and religious context in which they were created. Their geographical accuracy tells us much about the state of technology at the time of their creation.

Scholars who study these maps often want to take notes on certain maps or map regions, view certain areas in the context of today’s maps, associate map regions with historical events, places, or even persons. Annotations are the scholarly primitive that allows them to do that and to share their knowledge with others. However, the historic maps that are currently published on the Web, mostly by libraries, don’t allow scholars or end-users to annotate them in a way that is interoperable across systems.

1.1 Background

Maphub is a Web portal for hosting digitized, high-resolution historic maps. It allows users to annotate and contribute their knowledge about historic maps. It is currently being developed as part of the Open Annotation Collaboration demonstration activities. All user-contributed annotations are openly shared on the Web and represented in the W3C Open Annotation data model. Thereby they become Web resources that can be accessed by various clients simply by dereferencing HTTP URIs.

The current demo release is bootstrapped with approximately 6000 public domain high-resolution map scans from the Library of Congress Historic Map division. A first concept screencast and the portal's source code is available. The portal is built using Ruby on Rails and reuses components (and algorithms) that were previously developed for the YUMA annotation framework. User-contributed annotations are linked with resources in other publicly available datasources, such as DBpedia and GeoNames.

1.2 Example

In the following examples we refer to a high-resolution scan of the Waldseemüller map. We keep a copy of the JPEG2000 digitized map file from the LoC Maps division at http://samos.mminf.univie.ac.at/maps/raw/g3200.ct000725C.jp2. The raw map image has a width of 18000 and a height of 10029 pixels.

Figure 1. Example Map.

Map images are usually converted into tile sets for online viewing and zooming. MapHub uses the magicktiler tool to create Zoomify tilesets from the original JP2 images. The tilesets now reside parallel to the original image files, e.g.: http://samos.mminf.univie.ac.at/maps/ts_zoomify/g3200.ct000725C.jp2. But since tilesets represent a special view of the original map image, we can assume that the target of an annotation is always the original full-resolution map image file.

2. Georeference Annotations

A control points associates a place URI (a Semantic Tag) with a point on the map, which is the annotation (Target).

Figure 2. Example Control Point.

Place URIs are provided by the Geonames online gazetteer:

  • http://sws.geonames.org/2643743/: London, UK
  • http://sws.geonames.org/2517117/: Granada, ES
  • http://sws.geonames.org/2761369/: Vienna, AT

Georeference Annotations (Control Points) in Maphub are dereferenceable Web resources. When a client issues an HTTP GET request against the Georeference Annotation HTTP URI, Maphub determines the response format based on the value of the HTTP Accept header submitted by the client.

Figure 2 shows an example Georeference Annotation (http://maphubdev.mminf.univie.ac.at/control_points/1) represented in the Open Annotation model.

3. Commentarial Annotations

Commenting on maps or map segments is another frequent historic map annotation use case. This form of annotation varies in technical complexity, depending on the body structure and the annotation target. In the following, we illustrate how to represent various forms of commentarial annotations.

Associating unstructured text with a whole map is the simplest use case. The body is represented as inline information, the target is the map image file.

Figure 6. Simple Map Comment
Model Instance
Maphub Simple Map Comment - Turtle serialization

<http://maphub.info/annotation/1>
    a oac:Annotation ;
    oac:hasBody <urn:uuid:b70232d0-66f9-11e1-b86c-0800200c9a66> ;
    oac:hasTarget <http://samos.mminf.univie.ac.at/maps/g3400.ar077400.jp2> .

<urn:uuid:b70232d0-66f9-11e1-b86c-0800200c9a66>
    a cnt:ContentAsText ;
    cnt:characterEncoding "UTF-8"^^<http://www.w3.org/2001/XMLSchema#string> ;
    cnt:chars "One of these lakes must be Cayuga Lake..."^^<http://www.w3.org/2001/XMLSchema#string> .

We should consider allowing literals as body values, as in the Annotation Ontology. This is probably the simplest case and character encoding doesn't need to be specified because this the serialization format's job. Turtle, for instance, requires all data to be encoded in UTF-8.

3.3. Simple Comment on a Non-Rectangular Map Segment

Non-rectangular map segments, such as polygons, cannot be expressed by the W3C Media Fragment URI specification. Therefore, we introduce an intermediary ConstraintTarget (CT-1) node, which constrains the target of the annotation to a particular segment. The segment information is further describe by a Constraint resource (C-1). Since we want to use SVG to express segment information, the Constraint node is of type oac:SVGConstraint, which is a subclass of oac:Constraint. The SVG snippet describing the segment is an XML document, which is attached as literal value of type rdf:XMLLiteral to the Constraint node. The SVG literal must be formatted as canonical XML.

3.4. Complex Comments

Comments often go beyond simple textual descriptions. In the Maphub portal, for instance, the system proposes semantic tags (e.g., DBPedia URIs) to the user while they are commenting on a certain map or map segment. When users accept proposed tags by selecting them, they become part of the annotation. We can represent this by using the Annotation Ontology's Tagging property ao:hasTopic.

Note (RSi): +1 for the hasTopic approach! (Do you know if this - or similar - will be part of the consolidated OAC/AO draft?)

    a cnt:ContentAsText ;
    cnt:characterEncoding "UTF-8"^^<http://www.w3.org/2001/XMLSchema#string> ;
    cnt:chars "One of these lakes must be Cayuga Lake..."^^<http://www.w3.org/2001/XMLSchema#string> .

In case we want to mark up named entities (such as mentions of places or persons) directly in the annotation text, we propose to express the annotation as a segment of TEI encoded text. (Please note that the inline markup itself IS NOT a valid TEI document. However we argue it is straightforward to compile a TEI document for the annotated image through concatenation of annotations.)

Annotation Text with TEI Markup Example

<http://maphub.info/annotation/4>
    a oac:Annotation ;
    oac:hasBody <urn:uuid:b70232d0-66f9-11e1-b86c-0800200c9a66> ;
    oac:hasTarget <http://samos.mminf.univie.ac.at/maps/g3400.ar077400.jp2> ;
    
<urn:uuid:b70232d0-66f9-11e1-b86c-0800200c9a66>
    a cnt:ContentAsXML ;
    cnt:characterEncoding "UTF-8"^^<http://www.w3.org/2001/XMLSchema#string> ;
    cnt:chars "<text>
<body>
<p>One of these lakes must be 
  <placeName>
    <region>Cayuga Lake</region>
    <idno type="URL">http://dbpedia.org/resource/Cayuga_Lake</idno>
  </placeName>
  ...
</p>
</body>
</text>"^^<http://www.w3.org/2001/XMLSchema#string> .

Note: If we want to represent the relationships between the tags and the text (e.g., the named entity strings) things are getting more complex. We could probably model this as annotations on the textual body. But for Maphub I don't see any necessity.

Note (RSi): I was thinking along those lines for the future of yuma.js. However, the way I'd currently do it would be to use TEI-encoded text instead of plain text (that way tags would be embedded in the text).
